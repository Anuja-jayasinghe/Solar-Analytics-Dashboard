# CEB Step 1 Implementation Plan

## Document Input and Secure Saving

## 1. Objective

Implement the first production step of CEB bill automation:

- Accept CEB bill documents from admin users.
- Save files securely in private storage.
- Create a reliable ingestion record for each upload.
- Provide clear upload status and auditability.

This step does not perform OCR yet. It prepares the pipeline so OCR can be attached in the next step.

## 2. Scope

In scope:

- Admin upload UI in existing CEB admin area.
- Backend upload endpoint with admin authorization.
- Supabase Storage private bucket for bill files.
- Ingestion table insert per file upload.
- File hashing for deduplication readiness.
- Basic status lifecycle: `received`, `failed`.
- Error handling and observable logs.

Out of scope for this step:

- OCR extraction.
- Review queue UI.
- Email ingestion.
- Auto-approval logic.

## 3. Architecture for Step 1

1. Admin uploads file from web UI.
2. Request goes to protected serverless endpoint.
3. Endpoint validates file type/size.
4. Endpoint calculates `sha256` hash.
5. File is stored in private Supabase bucket (`ceb_bills`).
6. Endpoint inserts metadata row into `ceb_bill_ingestions`.
7. UI receives success response with ingestion id and file metadata.

## 4. Data Model (Step 1 Minimum)

### 4.1 Table: `ceb_bill_ingestions`

| Column | Type | Required | Notes |
|---|---|---|---|
| `id` | UUID PK | Yes | Generated by DB |
| `source_type` | text | Yes | `manual_upload` for this step |
| `file_path` | text | Yes | Storage object path |
| `file_sha256` | text | Yes | Raw file hash |
| `received_at` | timestamptz | Yes | Default `now()` |
| `status` | text | Yes | `received` or `failed` |
| `error_message` | text nullable | No | Error detail when failed |
| `created_by` | text nullable | No | Admin id/email |

Recommended indexes:

- Index on `received_at` desc for recent activity listing.
- Index on `status` for monitoring.
- Optional unique index on `file_sha256` (or soft dedupe check first, strict unique later).

## 5. Storage Design

Bucket:

- Name: `ceb_bills`
- Visibility: private
- Path convention:
  - `ceb/{yyyy}/{mm}/{adminId}/{timestamp}_{safeFileName}.pdf`

Accepted file types:

- `application/pdf`
- `image/png`
- `image/jpeg`

File limits:

- Max single file size: 10 MB initially.
- Reject archives or unsupported formats.

Security:

- No public URLs.
- Use signed URL only when preview is needed by authorized admin users.

## 6. API Design

### 6.1 Endpoint

- Method: `POST`
- Path: `/api/ceb-bills/upload`
- Auth: admin-only (reuse existing admin token middleware)
- Content type: multipart form-data

### 6.2 Request

Fields:

- `file`: required (pdf/png/jpeg)
- `source_type`: optional, default `manual_upload`

### 6.3 Response

Success (`201`):

- `ingestionId`
- `filePath`
- `fileSha256`
- `status` (`received`)
- `receivedAt`

Failure examples:

- `400`: invalid file type/size
- `401/403`: unauthorized/non-admin
- `409`: duplicate file hash (if dedupe enabled)
- `500`: storage or database write failure

### 6.4 Idempotency and Retry

For this step:

- UI should prevent double-submit while upload in progress.
- Backend should handle duplicate uploads gracefully via hash check.
- Return explicit duplicate status instead of generic failure where possible.

## 7. Frontend Changes

Primary integration point:

- Existing CEB admin page upload panel in `src/components/admin/CebDataManagement/index.jsx`.

Additions:

- Upload control (drag/drop + browse).
- Allowed file type and size hints.
- Progress indicator.
- Success state with ingestion id.
- Error state with actionable message.

Behavior:

- Keep existing manual entry fully available.
- Upload should be independent and non-blocking to existing form.

## 8. Backend Changes

Target files/services:

- New endpoint file: `api/ceb-bills/upload.js`
- Reuse middleware: `api/middleware/verifyAdminToken.js`
- Shared utility (new): file hash helper and storage path builder.

Implementation checklist:

1. Parse multipart input safely.
2. Validate MIME type and max size.
3. Compute SHA-256 from file buffer.
4. Upload to Supabase Storage private bucket.
5. Insert ingestion row.
6. Return structured response.
7. Log trace id + ingestion id for support.

## 9. Validation Rules (Step 1)

Hard validations:

- File required.
- MIME type in allow-list.
- File size under limit.

Soft validations:

- Filename normalization (remove unsafe chars).
- Optional duplicate warning if same hash exists.

## 10. Security and Compliance

- Admin-only endpoint access.
- Private bucket + strict policies.
- Avoid logging raw file content.
- Log actor (`created_by`) and timestamps for audit trail.
- Keep secrets in env vars only.

## 11. Observability and Operations

Log fields per request:

- request id
- admin id
- filename
- file size
- mime type
- storage path
- sha256
- ingestion id
- result status
- latency ms

Minimal operational checks:

- Upload success rate
- Average upload latency
- Count of failed ingestions

## 12. Acceptance Criteria

Step 1 is complete when:

1. Admin can upload supported file types from UI.
2. File is saved in private bucket with expected path convention.
3. Ingestion row is created with `received` status.
4. Unauthorized users cannot upload.
5. Invalid file types/sizes are rejected with clear messages.
6. Existing CEB manual workflow remains unchanged.

## 13. Delivery Plan (Practical Sequence)

1. Create DB migration for `ceb_bill_ingestions`.
2. Create bucket and storage policies.
3. Build backend upload endpoint.
4. Integrate upload UI on admin page.
5. Add logs and error mapping.
6. Run test checklist and release behind admin-only access.

## 14. Test Checklist

Functional:

- Upload valid PDF -> success + ingestion created.
- Upload png/jpg -> success.
- Upload unsupported file -> blocked.
- Upload > 10 MB -> blocked.

Security:

- Non-admin upload attempt -> denied.
- Public file URL access -> denied.

Reliability:

- Retry same file twice -> deterministic behavior.
- Simulated storage failure -> status/error response recorded.

## 15. Risks and Mitigations

1. Large files cause timeout or memory pressure.
- Mitigation: strict size limit + stream/multipart controls.

2. Duplicate uploads from user double-clicks.
- Mitigation: UI disable during upload + hash dedupe check.

3. Future OCR coupling delays this step.
- Mitigation: keep Step 1 endpoint clean and OCR-agnostic.

## 16. Handover Notes for Team

- This step is foundation only. Do not include OCR in this release.
- Keep schema and endpoint naming aligned with the full automation plan.
- Preserve backward compatibility with current CEB manual data entry.
